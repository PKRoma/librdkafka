# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

librdkafka is a C implementation of the Apache Kafka client protocol (producer, consumer, admin), with a C++ wrapper in [src-cpp/](src-cpp/). The public C API is in [src/rdkafka.h](src/rdkafka.h); everything else in `src/` is internal. API/ABI stability is a hard rule — see [CONTRIBUTING.md](CONTRIBUTING.md#changes-to-existing-apis).

## Build commands

librdkafka uses a custom `mklove` configure system (not autotools, not plain CMake for the supported path):

```bash
./configure                                 # or ./configure --install-deps
make                                        # builds src/ and src-cpp/
make -C tests build                         # build the test runner
./dev-conf.sh [asan|tsan|ubsan|clean]       # rebuild with dev asserts + sanitizer
```

`CONFIGURATION.md` is regenerated by `make` from `examples/rdkafka_performance -X list`; don't hand-edit it.

## Running tests

The test suite lives under [tests/](tests/) — files named `0NNN-*.c` / `.cpp`. A test runner binary (`test-runner`) dispatches by number.

```bash
cd tests
make                          # parallel run against a cluster from test.conf / trivup
make quick                    # condensed CI-style run
make run_seq                  # one test at a time (easier to debug)
make run_local_quick          # only broker-less tests (fastest, no cluster needed)
TESTS=0103 make               # run one test (variable is TESTS, not TEST)
TESTS=0103 SUBTESTS=foo make  # limit to sub-tests matching substring
TESTS=0061 TEST_DEBUG=cgrp,fetch make     # enable client debug contexts
TESTS=0009 ./run-test.sh valgrind|helgrind|gdb
./until-fail.sh bare          # loop until a test fails (hunting flakes)
RD_UT_TEST=name TESTS=0000 make   # run a single unit test from rdunittest.c
```

Tests 0000–0999 are automated; 1000+ need manual setup; 8000+ are manual-only. Tests flagged `TEST_F_LOCAL` don't need a broker.

When adding a test, copy the next free `0NNN-*` file and register it in [tests/CMakeLists.txt](tests/CMakeLists.txt), [win32/tests/tests.vcxproj](win32/tests/tests.vcxproj), and two spots in [tests/test.c](tests/test.c) (the Makefile picks files up automatically, but the other three don't). See [tests/README.md](tests/README.md) for the full guide.

A trivup-based local cluster can be spun up with:
```bash
python3 -m pip install -U -r tests/requirements.txt
python3 -m trivup.clusters.KafkaCluster 3.4.0   # then `make` inside the shell
```

## Style & linting

clang-format for C/C++, flake8/autopep8 for Python. **Must run before committing:**

```bash
make style-check-changed      # verify
make style-fix-changed        # auto-fix
```

C style notes that clang-format doesn't catch (see [CONTRIBUTING.md](CONTRIBUTING.md#librdkafka-c-style-and-naming-guide)):
- Minimum standard is **gnu90** — no in-line variable declarations; declare at the head of a scope.
- 8-space indent, `/* */` comments only (no `//`), doxygen for function docstrings.
- Symbols are `rd_kafka_<subsystem>_<action>`; variable names mirror the type prefix (`rd_kafka_broker_t` → `rkb`).
- Protocol request names use Apache Kafka CamelCase (`rd_kafka_ProduceRequest`, `ErrorCode`).
- Don't null-check internal function arguments — test coverage handles misuse. Only validate at the public API boundary.
- Avoid `#ifdef`s in code; platform detection belongs in `configure.librdkafka`/mklove.

## Architecture

librdkafka is **multi-threaded with message-passing between threads via op queues** (`rd_kafka_q_t`, defined in [src/rdkafka_queue.h](src/rdkafka_queue.h)). Almost nothing is done synchronously on the application thread. Understanding the thread layout is the single most important thing for making non-trivial changes:

- **Main thread (app)**: where the app calls `rd_kafka_produce`, `rd_kafka_poll`, etc. Poll dispatches callbacks from the main queue.
- **Broker threads**: one per broker connection. Handle the Kafka wire protocol — serialize requests from `rd_kafka_buf_t` ([rdkafka_buf.c](src/rdkafka_buf.c)), read responses, dispatch replies back via ops. Defined in [rdkafka_broker.c](src/rdkafka_broker.c).
- **cgrp thread** (logically, runs on a broker thread): the consumer group coordinator state machine — joins, rebalances, offset commits. See [rdkafka_cgrp.c](src/rdkafka_cgrp.c).
- **Timer thread**: fires scheduled callbacks ([rdkafka_timer.c](src/rdkafka_timer.c)).
- **Background thread** (optional): async admin ops, etc. ([rdkafka_background.c](src/rdkafka_background.c)).

Locking conventions are documented per-struct in the header files (`rdkafka_int.h`, `rdkafka_broker.h`, etc.). When adding a field, document which lock guards it. TSAN (`./dev-conf.sh tsan`) is the primary tool for verifying lock correctness.

### Key subsystems and where they live

- **Producer path**: app → [rdkafka_msg.c](src/rdkafka_msg.c) → topic partition queue ([rdkafka_partition.c](src/rdkafka_partition.c)) → broker thread batches via [rdkafka_msgset_writer.c](src/rdkafka_msgset_writer.c) → ProduceRequest. Idempotence state in [rdkafka_idempotence.c](src/rdkafka_idempotence.c); transactions in [rdkafka_txnmgr.c](src/rdkafka_txnmgr.c).
- **Consumer path**: FetchRequest issued per broker ([rdkafka_fetcher.c](src/rdkafka_fetcher.c)) → response parsed by [rdkafka_msgset_reader.c](src/rdkafka_msgset_reader.c) → messages land on the partition's fetch queue → forwarded to the consumer group queue → app polls.
- **Consumer groups**: classic protocol and KIP-848 "next gen" both live in [rdkafka_cgrp.c](src/rdkafka_cgrp.c); assignors in [rdkafka_*_assignor.c](src/); assignment state in [rdkafka_assignment.c](src/rdkafka_assignment.c).
- **Admin API**: request builders and result types in [rdkafka_admin.c](src/rdkafka_admin.c) / [rdkafka_admin.h](src/rdkafka_admin.h); each admin op has its own state machine. Results are delivered via an `rd_kafka_queue_t` the caller provides.
- **Metadata**: fetching in [rdkafka_metadata.c](src/rdkafka_metadata.c), cache in [rdkafka_metadata_cache.c](src/rdkafka_metadata_cache.c).
- **Protocol**: request/response serialization in [rdkafka_request.c](src/rdkafka_request.c); wire format primitives in [rdkafka_buf.c](src/rdkafka_buf.c); protocol constants in [rdkafka_protocol.h](src/rdkafka_protocol.h).
- **Transport**: [rdkafka_transport.c](src/rdkafka_transport.c) (sockets + poll/epoll/kqueue), [rdkafka_ssl.c](src/rdkafka_ssl.c), SASL dispatcher in [rdkafka_sasl.c](src/rdkafka_sasl.c) with mechanism-specific files (`rdkafka_sasl_{plain,scram,oauthbearer,cyrus}.c`).
- **Mock cluster**: [rdkafka_mock.c](src/rdkafka_mock.c), [rdkafka_mock_handlers.c](src/rdkafka_mock_handlers.c), [rdkafka_mock_cgrp.c](src/rdkafka_mock_cgrp.c) — an in-process broker used by `*_mock.c` tests for deterministic failure injection without a real cluster.
- **Telemetry**: OpenTelemetry encoder/decoder in [rdkafka_telemetry*.c](src/). nanopb + OTel `.proto` bindings live under [src/nanopb/](src/nanopb/) and [src/opentelemetry/](src/opentelemetry/); they are generated — regenerate via [src/generate_proto.sh](src/generate_proto.sh) rather than editing `.pb.c` files.

### Ops and queues

An `rd_kafka_op_t` is the unit of inter-thread communication. A thread enqueues an op on a target queue; the destination thread drains the queue in its run loop. This is why most APIs have both a sync and async form — the sync form just does an op dispatch + wait. When adding cross-thread behaviour, prefer extending the op mechanism over adding a new mutex. See [rdkafka_op.h](src/rdkafka_op.h) for op types and [rdkafka_queue.h](src/rdkafka_queue.h) for the queue primitives (including forwarding, which is how the app's consumer queue can redirect to a custom one).

## Contribution gates

- New public APIs cannot be changed after merge — see [CONTRIBUTING.md](CONTRIBUTING.md#new-public-apis). No public structs; use opaque types with accessors. Prefer async APIs returning an `rd_kafka_queue_t`.
- User-visible changes must have a one-line entry in [CHANGELOG.md](CHANGELOG.md) under the appropriate section.
- Bug fixes should add a regression test in [tests/](tests/).
- Keep the library portable — no new dependencies unless also available via vcpkg and buildable through mklove.
